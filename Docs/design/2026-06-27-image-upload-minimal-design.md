# 程序说明

本文档记录 Windows-Copilot-API 项目中“图片上传”能力的最小设计，目标是在当前已可用的文本聊天基线上，先支持 OpenAI 风格图片消息，并通过已验证稳定的浏览器会话完成真实页面上传与问答。

## 背景

当前项目的文本能力已恢复，但图片输入尚未可用，主要原因有两点：

- `server/api.py` 目前只把消息转成纯文本 prompt，没有提取图片内容
- `copilot/driver.py` 在收到 `image` 参数时直接抛错，不允许图片链路继续执行

由于新 ChatHub 纯 HTTP 协议尚未完全稳定，当前文本能力已经通过“纯 HTTP 优先、浏览器兜底”恢复。图片上传若继续走纯 HTTP 逆向，风险明显更高，因此首版选择直接走浏览器真实页面上传。

## 目标

- 支持 `/v1/chat/completions` 接收 OpenAI 风格图片消息
- 支持单张本地图片 + 文本提示词的问答
- 保持现有文本聊天能力不回退
- 将图片消息转换为浏览器上传路径，复用当前项目登录态和持久化 Chrome 会话

## 非目标

- 不做纯 HTTP 图片上传协议逆向
- 不支持多张图片
- 不支持远程 URL 下载图片后再上传
- 不支持图片生成能力
- 不扩展 `/v1/images/*` 一类额外接口

## 输入约束

首版仅支持以下 OpenAI 风格消息子集：

- `messages[].content` 为数组
- 数组中包含一段 `type=text`
- 数组中包含一段 `type=image_url`
- `image_url.url` 必须是本地文件路径，暂不接受远程 URL 和 base64 data URL

若消息中有多张图，则直接报错，避免隐式选择导致行为不确定。

## 方案选择

### 方案 A：浏览器上传

在真实 Copilot 页面中复用现有持久化会话，自动选择本地图片文件并发送文本。

优点：

- 与当前已可用文本兜底路径一致
- 不依赖继续逆向新协议中的上传接口
- 成功率最高，实施成本最低

缺点：

- 需要浏览器页面具备上传控件
- 相比纯 HTTP 更慢

### 方案 B：纯 HTTP 上传

继续逆向新协议，找到图片上传端点和消息格式，再从驱动层直接发送。

优点：

- 理论上更适合后端纯无头部署

缺点：

- 当前协议仍在变化
- 成本高，失败风险高

### 方案 C：双通道

优先纯 HTTP，失败后退回浏览器上传。

优点：

- 理论兼容性更强

缺点：

- 实现复杂，当前收益不高

结论：采用方案 A。

## 设计

### 1. API 层

修改 `server/schemas.py` 与 `server/prompt.py` 附近逻辑，支持从 OpenAI 风格消息中解析：

- 文本 prompt
- 单张本地图片路径

`server/api.py` 继续保持当前路由不变，只是在调用 `client.chat()` / `client.stream()` 时，将解析出的图片路径一并传下去。

### 2. Client 层

`copilot/client.py` 继续保持对外接口稳定，只新增向下透传 `image` 参数，不改变返回结构。

### 3. Driver 层

`copilot/driver.py` 中，当检测到 `image` 参数时，不再报“暂不支持图片上传”，而是直接走浏览器路径：

- 启动或复用 `BrowserCopilot`
- 在真实页面里上传图片
- 发送用户文本
- 流式读取回复

文本聊天原有逻辑保持不变：

- 纯 HTTP 新协议优先
- 升级失败时浏览器兜底

### 4. Browser 层

在 `copilot/browser.py` 新增最小图片发送方法：

- 打开当前 Copilot 聊天页
- 定位上传按钮或文件输入控件
- 选择本地图片文件
- 输入文本 prompt
- 发送消息
- 复用已有 WebSocket 监听逻辑读取机器人回复

若页面元素定位失败，则抛出清晰异常。

### 5. 错误处理

所有图片相关失败都必须返回明确错误：

- 图片路径不存在
- 输入为远程 URL / base64，当前不支持
- 多图输入，当前不支持
- 页面未找到上传控件
- 上传后超时无回复

## 验证计划

按以下顺序验证：

1. 使用一张本地图片和一句简单问题，验证 `client.chat(..., image=...)` 可返回文本
2. 使用 `/v1/chat/completions` 的 OpenAI 风格图片消息验证非流式返回
3. 使用 `/v1/chat/completions` 的 OpenAI 风格图片消息验证流式返回
4. 验证纯文本请求仍保持可用

## 风险

- Copilot 页面上传控件可能因页面改版而变更选择器
- 浏览器上传链路速度可能慢于纯文本
- 如果页面在某些节点下临时关闭图片入口，本方案会受影响

## 回滚方式

若图片上传实现不稳定，可仅回滚：

- `server/schemas.py`
- `server/prompt.py`
- `server/api.py`
- `copilot/client.py`
- `copilot/browser.py`
- `copilot/driver.py`

回滚后不会影响当前已可用的文本聊天能力。
